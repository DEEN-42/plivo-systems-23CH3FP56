# NOTES.md — Design Summary

Our implementation uses a depth-2 interleaved XOR Forward Error Correction (FEC) scheme with group size k=3: the sender maintains two concurrent XOR accumulators and round-robins incoming 20 ms frames between them, so each group collects members spaced 2 frames apart (e.g., frames {0, 2, 4} form one group and {1, 3, 5} form another), and sends one 169-byte parity packet after every 3 data packets. The receiver buffers all arriving packets in a dynamically allocated flat array indexed by frame sequence number, runs XOR recovery as soon as k−1 data packets plus the parity packet for a group have arrived, and delivers frames to the harness player via a dedicated playout timer thread that wakes at `t0 + DELAY_MS + i×20 ms − 1 ms` for each frame i and never blocks waiting on a missing frame. For grading purposes, **our declared playout delay is 160 ms**, which is the minimum value that reliably accommodates the depth-2 interleaved group span (80 ms) plus the worst-case one-way relay delay (80 ms) seen in Profile B. The 160 ms figure also provides the safety margin needed for unknown grading profiles that may have higher loss rates or wider delay distributions than the profiles we tested against. Our system will fail validity if the network induces correlated burst losses of 3 or more consecutive frames, because each FEC group can correct at most 1 loss regardless of group structure, and three consecutive losses will hit the same interleaved group twice with no remaining parity to recover from. It will also fail if the end-to-end one-way relay delay regularly exceeds `DELAY_MS − 80 ms = 80 ms` on Profile B, as packets arriving after the playout deadline are treated as permanent misses and cannot be recovered retroactively. The feedback path (ports 47003/47004) is deliberately unused: any bytes sent downstream would be counted against the 2.0× overhead cap and would fund retransmissions that cannot arrive within the deadline budget given relay RTTs of 40–160 ms.
