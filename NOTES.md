# NOTES.md — Design Summary

Our transport architecture implements a lock-free, depth-2 interleaved XOR Forward Error Correction (FEC) scheme with a group size of k=2. The sender maintains two concurrent XOR accumulators, round-robining incoming 20 ms frames so that consecutive frames fall into separate groups (e.g., frames {0, 2} form one group and {1, 3} form another). This creates a 40 ms block span, allowing us to survive any two-frame consecutive burst loss while minimizing latency overhead and staying within a 1.58x bandwidth budget. The receiver buffers arriving packets in a dynamically allocated flat array, tracks duplicates using a bitset, and executes O(1) XOR recovery in the lock-free ingress thread immediately upon receiving exactly one data packet and one parity packet for a group. A dedicated playout timer thread wakes 5 ms before the exact deadline for each frame (`t0 + DELAY_MS + i×20 ms - 5 ms`) to account for OS scheduling variance and delivers the payload without ever blocking on missing data. For grading purposes, we declare a playout delay of 130 ms. This value perfectly accommodates our 40 ms block span plus the 80 ms worst-case network jitter observed in Profile B. The system will fail validity if the network introduces correlated burst losses of three or more consecutive frames, as the depth-2 interleave cannot protect against two hits to the same group. It will also fail if steady-state, one-way network delay strictly exceeds our 80 ms transit budget, as packets arriving after the playout deadline cannot be retroactively recovered. The feedback path (ports 47003/47004) remains deliberately unused, as downstream NACK retransmissions cannot mathematically complete a round trip within the deadline budget.
