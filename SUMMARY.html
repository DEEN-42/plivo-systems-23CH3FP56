Data Flow

  • The harness source emits 164-byte frames (4 B big-endian seq + 160 B payload) to the sender at exactly t₀ + i × 20 ms.
  • The sender wraps each frame in a 5-byte custom header (169 B total) and sends it upstream through the relay. It simultaneously maintains two interleaved XOR accumulators, emitting one FEC parity packet for every three data packets.
  • The hostile relay applies loss, variable delay, duplication, and reordering per the active profile. Every byte it receives (on ingress) is counted toward the bandwidth overhead score — including dropped packets.
  • The receiver runs two threads: an ingress thread that buffers all arriving packets and performs XOR recovery, and a playout thread that fires at each frame's deadline and delivers recovered frames to the player.
  • The harness player records the first arrival time of each frame. A frame is valid only if it arrives before t₀ + DELAY_MS + i × 20 ms with the correct payload.
ℹ️
Feedback path intentionally unused. Ports 47003/47004 are left idle. Every byte sent downstream counts against the 2.0× overhead budget, and retransmission RTTs (40–160 ms through the relay) cannot fit inside the 160 ms playout window.
📦 Wire Format

Every packet transmitted from sender (port 47001) to receiver (port 47002) is exactly 169 bytes: a 5-byte custom header followed by the 164-byte payload.

🔧 Loss Recovery — FEC Design
⚡
Why FEC instead of ARQ (retransmission)?
ARQ requires a NACK packet to travel downstream through the relay (20–80 ms on Profile B, and itself subject to 5% loss), followed by the retransmission travelling upstream (another 20–80 ms). This means the fastest possible retransmission takes at least 40 ms round-trip and up to 160 ms — equal to or exceeding our entire playout budget. Every NACK byte also counts against the 2.0× overhead cap. FEC parity packets travel the same one-way path as data, adding zero latency, and their overhead is fully determined at design time.

Depth-2 Interleaved XOR FEC (k = 3)

The sender maintains D = 2 concurrent XOR accumulators and round-robins frames between them. With group size k = 3, a group completes every 6 frames and one parity packet is sent immediately after the third member arrives at the sender.

Frames are grouped so that consecutive frames always fall in different groups:

Group A: frames { 0, 2, 4 }  →  parity_A = frame_0 ⊕ frame_2 ⊕ frame_4
Group B: frames { 1, 3, 5 }  →  parity_B = frame_1 ⊕ frame_3 ⊕ frame_5
Group C: frames { 6, 8, 10 } →  ...
...
A burst of 2 consecutive losses (e.g. frames 4 & 5) hits Group A once and Group B once.
Each group loses only one member → both fully recoverable via XOR.

Recovery rule: if exactly one of the k data packets is missing and the parity packet has arrived, the receiver reconstructs the lost payload as lost = parity ⊕ (all other data packets).

📐
Delay cost of depth-2 interleaving. A consecutive grouping (k=3) would require delay_ms ≥ (k-1)×20 + max_relay_delay = 40 + 40 = 80 ms on Profile A. The depth-2 interleave spans (k-1)×D×20 = 80 ms, raising the minimum to 120 ms (A) / 160 ms (B) — a fixed +40 ms penalty in exchange for surviving any 2-frame consecutive burst. Dropping to k=2 consecutive would not help: a 2-frame burst still knocks out the entire group. Interleaving is the only structural defence.
⏱ Jitter Buffer & Playout

Memory Layout

The jitter buffer is a flat array of 164-byte slots, allocated dynamically at startup from DURATION_S: n_frames = (int)(duration_s × 1000 / 20) + 16. Each slot contains a 164-byte payload and an atomic_int present flag. Direct indexing by frame_seq avoids any ring-buffer modular arithmetic.

A bitset (uint64_t[]) tracks seen sequence numbers. Before processing any arriving packet, the receiver checks this bitset and silently discards relay-generated duplicates.

Two-Thread Design

  • Ingress thread — loops on recvfrom(:47002). Parses the 5-byte header, bounds-checks frame_seq, copies the payload into jbuf[frame_seq], sets present = 1 (with memory_order_release), and updates FEC group state. If a group now has k−1 data packets plus a parity packet, it XOR-recovers the missing frame and injects it into the jitter buffer — potentially before the playout deadline.
  • Playout timer thread — iterates frame indices i = 0 … n_frames−1. For each i it calls clock_nanosleep(CLOCK_REALTIME, TIMER_ABSTIME, …) targeting t₀ + DELAY_MS/1000 + i×0.020 − 0.001 seconds (1 ms safety margin). On wake: if jbuf[i].present it sends 164 bytes to the player; otherwise it skips and continues immediately to i+1. It never blocks waiting for a specific missing frame, preventing cascading deadline failures.
